# 批量导入与材料清单开发文档

> 基于当前仓库代码状态整理，更新时间：2026-04-24。
> 目标：在不牺牲现有使用舒适度的前提下，为系统新增“菜单批量导入”和“随机结果材料清单”两项能力，并同步补齐移动端与交互层约束。

## 1. 需求结论

已确认的产品决策如下：

- 批量导入首版使用 Excel 模板
- 导入时遇到同名菜单，默认跳过
- 历史记录中的材料清单，以抽取当时的快照为准
- 材料汇总遵循“能精确就合并，不能精确就分开展示，不伪造总量”

本文档基于以上结论继续展开，不再重复回到格式和原则层面。

## 2. 设计原则

这次功能落地必须遵守以下原则：

- 低打扰：新增功能不能打断现有“新增菜单”与“随机抽取”主流程
- 低学习成本：用户不需要先理解复杂字段模型，模板本身就能教会用户怎么填
- 移动端优先：导入预览和材料清单不能依赖桌面宽屏布局
- 渐进增强：首版先保证稳定和舒服，精细化换算、单位归一、图片批量导入不强行塞入首版
- 向后兼容：不破坏现有菜单数据、历史记录、备份导入导出与共享同步链路

## 3. 当前系统现状判断

结合现有仓库，新增功能有以下现实前提：

### 3.1 有利条件

- 菜单主数据当前保存在 Dexie，页面直接读本地库，批量导入可以优先在前端完成
- 菜单新增/编辑已有统一写入入口 `lib/space-ops.ts`
- 共享空间模式已经支持本地先写、后续 `pending` 同步，不需要为导入单独造一套同步协议
- `menu_items.ingredients` 与 `profile_roll_history.items` 都是 JSON 文本，不需要新增数据库列就能扩展结构
- 历史、备份、账号私有同步已经打通，后续只要扩展类型定义和序列化逻辑即可

### 3.2 现有短板

- `Ingredient` 当前只有 `name + amount`，适合录入，不适合稳定做数量汇总
- `RollHistory` 当前只保存抽中的菜名与菜单 ID，不保存材料快照
- 当前通用弹窗 `components/ui/modal.tsx` 更偏短内容，不适合“导入预览”和“长清单回看”这类移动端重交互场景
- [app/layout.tsx](/D:/kimi-workspace/Have_you_eaten_today/app/layout.tsx:18) 已移除 `maximumScale: 1`，避免影响移动端可访问性与舒适度

## 4. 功能一：菜单批量导入

### 4.1 目标

- 用户在菜单页可以直接下载 Excel 模板
- 用户按模板填写多道菜后上传
- 系统在本地解析、校验、预览后，再批量写入菜单
- 本地模式与共享空间模式都可用
- 对重复项、错误项、缺失标签有明确反馈

### 4.2 首版不做

- 不支持图片批量导入
- 不支持从任意第三方 Excel 自动猜字段
- 不支持 CSV / Markdown / TXT 多格式并行导入
- 不支持导入时覆盖已有菜单
- 不支持导入后自动创建组合模板或场景清单

### 4.3 推荐用户流程

菜单页新增一个低打扰入口，完整流程如下：

1. 用户进入“菜单管理”
2. 点击“批量导入”
3. 进入导入面板，可看到：
   - 下载模板
   - 格式说明
   - 选择文件
4. 选择 Excel 后，系统先在本地解析，不立即写库
5. 显示导入预览：
   - 可导入多少条
   - 因重名将跳过多少条
   - 有错误多少条
   - 将自动创建多少个新标签
6. 用户确认后执行导入
7. 导入完成后显示结果反馈：
   - 成功导入数
   - 跳过数
   - 错误数
   - 共享空间模式下显示“已加入同步队列”

### 4.4 入口与交互布局

### 菜单页入口

- 桌面端：放在菜单页头部操作区，与“新增菜单项”“批量管理”并列
- 移动端：仍放头部，但避免三颗等权重大按钮硬挤一排
- 建议布局：
  - 主按钮保留“新增菜单项”
  - “批量导入”作为次按钮
  - “批量管理”保持切换型按钮

### 导入面板形式

- 桌面端：居中大弹层
- 移动端：全屏或近全屏 Sheet，而不是小弹窗
- 原因：
  - 需要展示模板说明
  - 需要承载错误预览列表
  - 需要在小屏上处理长表格摘要

### 4.5 模板方案

首版采用程序生成的 `.xlsx` 模板，而不是静态文件。

理由：

- 模板字段与代码常量同源，不容易过期
- 后续字段改动时，不需要手工更新 `public` 下的模板文件
- 生成模板和解析模板都可以复用同一套字段定义
- 即使在 PWA / 离线模式下，也可以生成并下载模板

### 工作表结构

建议生成两个 sheet：

- `填写说明`
- `菜单数据`

### `菜单数据` 列设计

| 列名 | 必填 | 示例 | 说明 |
| --- | --- | --- | --- |
| 类型 | 是 | 菜谱 / 外卖 | 仅允许这两个值 |
| 名称 | 是 | 番茄炒蛋 | 菜单名称 |
| 菜系标签 | 否 | 家常菜, 川菜 | 多个标签用逗号分隔 |
| 类别标签 | 否 | 素菜, 汤 | 多个标签用逗号分隔 |
| 自定义标签 | 否 | 快手, 减脂 | 多个标签用逗号分隔 |
| 权重 | 否 | 3 | 不填默认 1，必须是 1-10 的整数 |
| 材料清单 | 否 | 见下文 | 仅菜谱使用 |
| 步骤 | 否 | 见下文 | 仅菜谱使用 |
| 心得 | 否 | 番茄选熟一点更好吃 | 仅菜谱使用 |
| 店铺 | 外卖必填 | 老乡鸡 | 仅外卖使用 |
| 店铺地址 | 否 | xx 路 xx 号 | 仅外卖使用 |

### 材料清单填写规则

每个单元格内一行一个材料，推荐格式如下：

```text
鸡蛋|3|个
番茄|2|个
盐|少许
生抽|2勺
```

解释：

- `材料名|数量|单位`：精确模式，系统可用于合并汇总
- `材料名|用量文本`：兼容模式，例如 `生抽|2勺`
- `材料名|少许`、`材料名|适量`：非精确模式，仅展示来源，不参与强行求和

这样设计的原因：

- 用户不需要额外维护“数量列 + 单位列 + 材料表”三层结构
- 愿意精细填写的用户，可以直接获得更好的材料汇总结果
- 不愿意精细填写的用户，仍然可以低成本录入

### 步骤填写规则

每个单元格内一行一个步骤，例如：

```text
番茄切块，鸡蛋打散
先炒鸡蛋盛出
再炒番茄出汁后回锅翻炒
```

首版不在模板里单独要求步骤时长，避免复杂化。

### 4.6 导入判重策略

首版默认按“类型 + 名称”判重，而不是只按名称。

原因：

- 同名菜可能同时存在“菜谱版本”和“外卖版本”
- 如果仅按名称判重，会误伤这类合法场景
- 这更符合现有系统“recipe / takeout”双类型的模型边界

具体规则：

- 若导入文件内出现相同 `类型 + 名称`，保留第一条，后续行标记为“文件内重复，已跳过”
- 若本地库里已存在相同 `类型 + 名称`，该行标记为“与现有菜单重名，已跳过”
- 跳过项不算错误，不阻塞其他行导入

### 4.7 标签处理策略

- `菜系标签 / 类别标签 / 自定义标签` 分别映射到现有 `TagType`
- 标签值支持中英文逗号分隔，导入前统一 trim
- 同一列内重复标签自动去重
- 若标签不存在，则自动创建对应类型标签
- 若同一文字分别出现在不同列，例如同名同时属于”类别”和”自定义”，按当前列类型分别创建或复用
- 首版按列类型分别创建标签（与现有系统 `name+type` 唯一性一致）
- **预览中展示”将新建的标签”完整列表（名称 + 类型）**，让用户确认
- 如果同一文字在不同类型下已存在，在预览中标注提示

### 4.8 校验规则

### 行级错误

以下情况定义为错误行，不导入：

- 类型为空或不是“菜谱 / 外卖”
- 名称为空
- 外卖行缺少店铺
- 权重不是 1-10 的整数或超出范围
- 材料行格式不合法，例如只有 `|2个` 没有材料名

### 行级警告

以下情况定义为警告或跳过，不阻塞其他行：

- 与现有菜单重名
- 文件内重复
- 菜谱没有材料
- 步骤为空

### 预览输出分类

导入预览页只需要把结果分成四类即可：

- 可导入
- 将跳过
- 有错误
- 新标签

不要在首版做大型表格编辑器，避免把“导入功能”做成“在线 Excel”。

### 4.9 技术方案

### 新增模块建议

- `components/menu-import-dialog.tsx`
- `lib/menu-import-template.ts`
- `lib/menu-import-parser.ts`
- `lib/menu-import-validator.ts`
- `lib/ingredient-parser.ts`

### 模块职责边界

- `ingredient-parser.ts`：处理 Excel 单元格文本（如 `鸡蛋|3|个`）→ `{ name, amount?, quantity?, unit? }`
- `menu-item-sanitize.ts` 的 `toIngredients()`：处理 JSON 反序列化（服务端同步数据）
- 两者输入格式不同，职责不重叠

### 依赖建议

- 新增 `exceljs`（MIT 许可证）作为 Excel 读写依赖，体积约 2MB
- 在导入弹层打开时通过 `dynamic import` 加载，避免影响日常首屏与随机页性能
- 生成模板和解析模板复用同一套字段定义

### 写库策略

导入确认后，采用**全有或全无（原子导入）**策略：

1. 预览阶段完成所有校验（行级错误/警告/重复），确保执行阶段只做写入
2. 执行阶段在**单个 Dexie 事务** `db.transaction("rw", [db.tags, db.menuItems, db.personalWeights], ...)` 中完成：
   - 创建本次导入缺失的标签
   - 创建菜单项
   - 写入非默认权重（≠1）的 `personalWeights` 记录
3. 如果事务中途异常，整体回滚，不做部分导入
4. 事务成功后，对非默认权重对应的菜单项调用 `scheduleProfileStateSync()` 标记个人权重同步
5. 若当前处于共享空间，菜单与标签的 `pending` 状态由 `space-ops` 自动处理

> **注意**：权重记录必须和菜单、标签在同一事务内写入，不能在菜单事务提交后再调用 `setWeight()`，否则权重写入失败会留下半截导入数据。

### 导入弹窗关闭行为

| 阶段 | 关闭行为 |
|------|----------|
| 选择文件 | 可关闭 |
| 解析中 | 可关闭，无副作用 |
| 预览确认 | 可关闭，丢弃预览数据 |
| 写库中 | **禁用关闭按钮**（Dexie 事务不能安全中断） |
| 结果反馈 | 可关闭 |

### 为什么不单独做服务端导入接口

- 当前应用是本地优先模型
- 导入结果本身需要先做本地预览
- 直接本地写库更符合 PWA / 离线能力
- 可以复用现有共享同步，不需要额外处理服务端导入事务、权限和回滚

### 4.10 体验细节

- 文件解析中显示明确 loading，不让用户怀疑页面卡死
- 预览统计数字使用 `tabular-nums`
- 移动端错误列表默认折叠，只展开前几项
- 提供”仅导入可用项”按钮，而不是要求用户修完所有问题才能继续
- 导入成功后给出”查看新增项”或自动刷新列表
- 共享空间下明确提示”导入成功，正在同步到空间”
- **多设备导入提示**：导入 UI 中提示”导入期间请避免在其他设备操作菜单”，因为去重只在单台设备本地生效，两台设备同时导入同名菜单同步后可能产生重复

### 4.11 风险与处理

### 风险 1：Excel 解析体积和移动端卡顿

处理：

- 使用动态导入（exceljs 在打开导入弹层时按需加载）
- 模板和解析逻辑复用同一套字段定义
- 首版只支持一个简单 sheet 结构，不做复杂样式和公式
- **文件大小硬限制**：文件不超过 5MB，行数不超过 500 行
- **软限制**：超过 100 行给提示但不阻止
- 文件选择后、解析前先检查文件大小，超硬限制直接拒绝并提示

### 风险 2：用户不会填写材料格式

处理：

- 在 `填写说明` sheet 提供完整示例
- 支持 `材料名|用量文本` 的宽松模式
- 不把首版设计成强结构三列表

### 风险 3：重复判定引起误解

处理：

- 在预览里清楚显示”按 类型 + 名称 判重”
- 将被跳过的行单独列出

### 风险 4：多设备同时导入冲突

处理：

- 去重只在单台设备本地生效
- 两台设备同时导入同名菜单，同步后可能产生重复
- **首版处理**：导入 UI 中提示”导入期间请避免在其他设备操作菜单”
- 后续版本可在服务端增加 name+kind 唯一性约束

### 风险 5：导入中途失败导致数据不一致

处理：

- 采用**全有或全无**策略，执行阶段在单个 Dexie 事务中完成
- 如果事务中途异常，整体回滚，不做部分导入
- 写库进行中禁用弹窗关闭按钮，防止中断事务

## 5. 功能二：随机结果材料清单

### 5.1 目标

- 随机抽取完成后，用户可以一键查看材料清单
- 历史记录中的每次抽取结果都可以回看材料清单
- 清单以抽取当时的菜谱快照为准，而不是后改后的菜单内容
- 汇总遵循“能精确就合并，不能精确就分开”

### 5.2 首版不做

- 不做跨单位换算，例如 `500g` 自动转 `0.5kg`
- 不做食材别名归一，例如“西红柿”和“番茄”自动视为同一种
- 不做购物车、外部采购平台联动
- 不做按人数自动倍增份量

### 5.3 用户流程

### 随机页

随机完成后，在结果卡片中新增二级操作：

- 再抽一次
- 查看材料清单

当结果中没有菜谱时：

- 不展示该按钮，或显示禁用态并说明“当前结果没有菜谱材料”

### 历史页

每条历史记录卡片新增入口：

- 查看材料清单

不要求用户先点进菜谱详情，再自己逐条整理。

### 首页最近抽中卡片

这不是必须项，但属于低成本体验增强：

- 首页最近抽中卡片可以复用历史记录的材料清单入口

### 5.4 数据结构调整

### `Ingredient` 扩展为“显示值 + 可选结构化值”

建议从：

```ts
interface Ingredient {
  name: string;
  amount?: string;
}
```

扩展为：

```ts
interface Ingredient {
  name: string;
  amount?: string;
  quantity?: number;
  unit?: string;
}
```

设计含义：

- `amount` 继续作为用户可见原始展示值
- `quantity + unit` 仅在系统能可靠解析或导入时写入
- 不能解析时不强求这两个字段存在

这样做的好处：

- 手工新增菜单时，用户依然只填“材料 + 用量”
- 首版 UI 不需要增加额外输入控件
- 后端与备份结构有了未来精细化的演进空间

### `RolledItem` 增加菜谱快照

建议扩展为：

```ts
interface RolledItem {
  menuItemId: string;
  name: string;
  kind: MenuItemKind;
  shop?: string;
  ingredientSnapshot?: Ingredient[];
}
```

说明：

- 菜谱项在写入历史时，把当时的 `ingredients` 一并存入 `ingredientSnapshot`
- 外卖项不写该字段
- 历史回看时优先使用 `ingredientSnapshot`
- 老历史记录没有该字段时，再回退到当前 `menuItems`

### 为什么不新建历史汇总表

- `rollHistory.items` 本身就是 JSON，可直接承载快照
- 这类数据天然跟某次抽取结果绑定，不需要独立维护关系表
- 现有 `profile-state`、备份导出、服务端 `profile_roll_history.items` 都是整段 JSON 存储，兼容成本更低

### 5.5 历史兼容策略

### 新历史

- 抽取发生时直接写入 `ingredientSnapshot`
- 后续即使原菜单被编辑或删除，历史仍能稳定展示当时材料

### 老历史

- 若 `ingredientSnapshot` 缺失，则尝试按 `menuItemId` 回查当前菜单
- 若菜单已被删除或材料为空，则在材料清单中显示“部分旧记录缺少快照，无法完整回溯”

该策略的优点是：

- 无需一次性迁移旧历史
- 首版可以平滑上线
- 风险小于“强制补历史快照”

### 5.6 材料汇总算法

### 处理范围

- 仅汇总 `kind === recipe` 的项
- `takeout` 结果不进入材料合并，只在界面中单独提示“本次结果含外卖，不生成采购材料”

### 精确合并条件

同一材料满足以下条件时可以合并：

- 材料名一致（经过轻量规范化）
- 单位一致
- 数量是单个确定数值

示例：

- `鸡蛋 3个 + 鸡蛋 2个 = 鸡蛋 5个`
- `五花肉 500g + 五花肉 300g = 五花肉 800g`

### 不合并条件

以下情况不做强行求和：

- 材料名不同
- 单位不同
- 用量为 `适量 / 少许 / 若干`
- 用量是范围值，例如 `2-3个`
- 无法解析出稳定数字

示例：

- `生抽 2勺` 与 `生抽 15ml` 分开展示
- `盐 少许` 只列入“需自行把握”

### 输出结构建议

清单页分为三段：

1. 精确汇总
2. 需分别准备
3. 非菜谱项说明

其中“需分别准备”每一项要带来源菜名，避免用户不知道这份材料对应哪道菜。

### 5.7 名称规范化策略

首版只做轻量规范化，不做语义归一：

- trim 首尾空格
- 折叠中间多余空格
- 拉丁字符转小写

不做：

- “番茄 = 西红柿”
- “小葱 = 葱花”
- “鸡蛋（去壳） = 鸡蛋”

原因：

- 这类语义规则容易误合并
- 首版目标是可靠，不是聪明

### 多人抽选场景的材料快照策略

多人抽选（`shouldUseSharedRoll = true`）由服务端推荐并返回 `RolledItem[]`，客户端写入本地历史。该场景有 4 个需要修改的位置：

| # | 文件 | 位置 | 说明 |
|---|------|------|------|
| 1 | `lib/roll.ts` | `rollSingle()` 构造 RolledItem 时 | 填充 ingredientSnapshot |
| 2 | `lib/roll.ts` | `rollCombo()` 构造 RolledItem 时 | 同上 |
| 3 | `lib/server-recommendations.ts` | `toRolledItem()` | 服务端需扩展返回 ingredientSnapshot |
| 4 | `app/random/page.tsx` | 多人抽选客户端写入历史前 | 用 menuItemId 从本地 menuItems 补填 |

**多人抽选快照补充流程**：

1. 客户端收到服务端结果后，遍历 `RolledItem[]`
2. 对每个 `kind === "recipe"` 的项，用 `menuItemId` 从本地 `db.menuItems` 查询 `ingredients`
3. 如果该菜不在本地（其他用户创建的），`ingredientSnapshot` 留空，回退到老历史策略
4. 服务端 `toRolledItem()` 也应扩展，从服务端 `menu_items` 表查 `ingredients` 一并返回

### 5.10 风险与处理

### 风险 1：历史快照会增大存储体积

处理：

- 只写入 `ingredientSnapshot`（材料），不写入步骤（steps）
- 材料清单只含材料，首版不扩展为 `recipeSnapshot`
- 如需展示步骤，用户可点击查看菜单详情
- 不把整道菜单的步骤、标签、图片一起塞进历史
- 历史本身已有 500 条上限，风险可控

### 风险 2：材料解析精度不稳定

处理：

- 解析成功才写 `quantity / unit`
- 解析失败保留 `amount` 原样展示
- 汇总逻辑不冒进，不把模糊值硬转成总量

### 风险 3：老历史无法完整回溯

处理：

- 新历史从上线起稳定带快照
- 老历史按当前菜单兜底
- UI 明确标注”旧记录可能不完整”

### 前向兼容性说明

- `ingredientSnapshot` 是新增的可选字段，不是索引字段
- 老版本客户端写入/读取不会崩溃（Dexie 忽略未知非索引字段）
- 老版本 `pushCurrentProfileState()` 会原样转发新字段，不会丢失
- **老版本无法使用材料快照功能**，需要在 changelog 中提示升级
